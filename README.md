# Quix Environment Operator

<!-- solution-docs:begin readme -->
A Kubernetes controller that provisions isolated application environments on demand. A cluster-scoped `Environment` custom resource is the request surface: the operator turns each one into a dedicated namespace with tightly scoped RBAC, so the Quix platform can manage workloads on customer-hosted clusters without ever holding cluster-admin.

**Note:** This project is open source and can be used by anyone, but it is primarily designed for use within the Quix ecosystem.

**Start here:** [docs/overview.md](docs/overview.md) — what problem this solves and what it deliberately does not do.

## Key features

- [Environment provisioning](docs/features/environment-provisioning.md) — declarative namespace creation with strict naming, ownership tracking, and controlled metadata propagation
- [Scoped platform RBAC](docs/features/scoped-platform-rbac.md) — per-namespace RoleBindings to a pre-approved ClusterRole, with privilege-escalation validation
- [Lifecycle & status](docs/features/lifecycle-and-status.md) — phase tracking, Kubernetes events, and finalizer-driven cleanup

Architecture and security model: [docs/architecture.md](docs/architecture.md) · Key design decisions: [docs/decisions.md](docs/decisions.md)

## Deployment

Packaged as a Helm chart for customer-managed Kubernetes clusters:

```bash
helm repo add quix-environment-operator https://quixio.github.io/quix-environment-operator/ && helm repo update
helm upgrade --install quix-environment-operator quix-environment-operator/quix-environment-operator \
  -n quix-operator --create-namespace
```

All configuration options (namespace suffix, environment-ID regex, service account and ClusterRole names, `pods/exec` gating, …) are documented in [values.yaml](deploy/quix-environment-operator/values.yaml), which is the authoritative reference.
