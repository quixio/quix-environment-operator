# Overview

<!-- solution-docs:begin overview -->
## The problem

The Quix platform deploys and manages customer workloads on Kubernetes. When the cluster is hosted by the customer (bring-your-own-cluster), the platform cannot be trusted with broad cluster access: the customer needs a reviewable, minimal permission boundary, and the platform needs a way to carve out isolated spaces for each application environment without a human in the loop.

This operator is that boundary. It watches a single cluster-scoped custom resource, `Environment`, and provisions exactly one namespace per environment together with the RBAC that lets the platform service account work **inside that namespace only**. Everything the platform can do on the cluster is expressed in Helm chart manifests the customer installs and can audit up front; the operator never grants anything beyond them at runtime.

## What it is

It is a cluster-local provisioning boundary for Quix environments. The platform asks for an environment through Kubernetes, and the operator turns that request into a namespace, scoped platform access, status, events, and ordered cleanup.

## What it is not

It is not the workload deployer: the platform still creates application resources inside the environment. It is not a general-purpose namespace factory: naming, metadata, and access rules are intentionally shaped around the Quix platform. It is not runtime governance for quotas, limits, or network policy inside each environment; customers apply those through their own cluster policy. It also does not manage multiple clusters from one instance.

## Honest boundaries

### Enforced

The namespace boundary is enforced by Kubernetes RBAC and by ownership checks before the operator adopts or deletes resources. Platform workload access is granted per environment namespace through RoleBindings to a reviewed ClusterRole. The operator validates that role for escalation-capable permissions before relying on it.

### Not a hard boundary

The platform’s permission set is still whatever the customer-approved ClusterRole allows inside an environment namespace. Additional subjects manually added to managed RoleBindings may be preserved by design. Workload runtime controls, audit policy, quotas, and network isolation are outside this operator’s hard boundary unless the cluster provides them separately.
