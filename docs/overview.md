# Overview

<!-- solution-docs:begin overview -->
## The problem

The Quix platform deploys and manages customer workloads on Kubernetes. When the cluster is hosted by the customer (bring-your-own-cluster), the platform cannot be trusted with broad cluster access: the customer needs a reviewable, minimal permission boundary, and the platform needs a way to carve out isolated spaces for each application environment without a human in the loop.

This operator is that boundary. It watches a single cluster-scoped custom resource, `Environment`, and provisions exactly one namespace per environment together with the RBAC that lets the platform service account work **inside that namespace only**. Everything the platform can do on the cluster is expressed in Helm chart manifests the customer installs and can audit up front; the operator never grants anything beyond them at runtime.

## How it works, in one paragraph

The platform creates an `Environment` object carrying an ID. The operator validates the ID, creates a namespace named from it plus a configurable suffix, labels it as operator-managed, and binds the pre-approved platform ClusterRole to the platform ServiceAccount in that namespace via a RoleBinding. Deleting the `Environment` tears the namespace down again — but only if the operator can prove it owns it. Status, phase, and Kubernetes events make every step auditable.

## Who it's for

- **Customers hosting clusters** — install the chart, review the two ClusterRoles it defines, and let the platform provision environments within those bounds.
- **Security reviewers** — the permission model is deliberately small and documented in [architecture.md](architecture.md) and [decisions.md](decisions.md).
- **Platform integration engineers** — the `Environment` CRD is the entire integration contract.

## Non-goals

- **Deploying workloads.** The operator creates the space and the access; the platform itself deploys pods, services, and everything else into the namespace.
- **Resource governance inside environments.** No quotas, limit ranges, or per-environment network policies are created; if the customer needs those, they apply them through cluster policy.
- **Managing the platform's permission set at runtime.** The platform ClusterRole ships in the Helm chart and changes only via chart upgrade; the operator validates it but never edits it.
- **General-purpose namespace provisioning.** Naming, labelling, and metadata rules are opinionated for the Quix platform (e.g. all propagated metadata keys must be `quix.io/`-prefixed).
- **Adopting pre-existing namespaces.** A namespace not created (and labelled) by the operator for that same environment is never touched, and collisions fail the environment rather than hijack the namespace.
- **Multi-cluster orchestration.** One operator instance manages one cluster.
