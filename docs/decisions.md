# Key design decisions

<!-- solution-docs:begin decisions -->
Lightweight ADR-style records of the non-obvious choices in this operator. Each is inferable from the code and chart manifests; consequences describe what the choice costs or enables.

## 1. The operator holds `bind`, not the permissions it grants

**Context.** Kubernetes escalation prevention requires that whoever creates a RoleBinding either already holds every permission the bound role grants, or holds the `bind` verb on that role. Mirroring the full platform permission set into the operator's own ClusterRole would satisfy this — and make the operator as powerful as the platform.

**Decision.** The operator ClusterRole is granted only `bind` on the named platform ClusterRole, plus the operational permissions it directly uses.

**Consequence.** A security reviewer reads the operator's footprint as "operational permissions + `bind` on one named role", not the union of workload permissions. The cost is indirection: the workload permission set lives solely in the platform ClusterRole, referenced by name.

## 2. Cluster-scoped `Environment` CRD, with one cluster-wide platform grant

**Context.** Environments are not namespace residents — they *create* namespaces — so the CRD is cluster-scoped. But a namespaced RoleBinding cannot grant access to a cluster-scoped resource, and the platform must be able to create `Environment` objects.

**Decision.** The platform ServiceAccount gets a dedicated ClusterRoleBinding to a narrow ClusterRole covering only `quix.io/environments`.

**Consequence.** The namespace-isolation guarantee applies to workload resources; `Environment` objects are the single, documented cluster-wide exception. Anyone auditing the chart should expect exactly one platform ClusterRoleBinding.

## 3. Fail-fast validation of the platform ClusterRole

**Context.** If the configured ClusterRole is missing or contains escalation-capable rules, every environment would fail at RoleBinding time — after namespaces were already created.

**Decision.** The ClusterRole is validated at operator startup (exit on failure, before any reconcile) *and* re-validated on every RoleBinding reconcile.

**Consequence.** Misconfiguration surfaces as a crash-looping operator pod rather than half-provisioned environments; runtime re-validation catches out-of-band edits at the cost of an extra read per reconcile.

## 4. Ownership by identity labels, verified before every destructive action

**Context.** Namespace names are derived from user-supplied IDs; a name collision with an existing namespace (or a stale one from another environment) must not lead to adoption or deletion of something the operator doesn't own.

**Decision.** Managed namespaces carry identity labels (managed-by, environment ID and name); the operator verifies them before adopting on create and again before deleting.

**Consequence.** A collision fails the `Environment` with a clear status instead of silently hijacking a namespace. Stripping the labels orphans the namespace from the operator's perspective — deliberate: safety over self-healing for identity.

## 5. RoleBinding subjects: tolerate additions, reset on drift of the expected subject

**Context.** Customers sometimes add extra subjects to managed RoleBindings out of band. Blindly overwriting subjects each reconcile destroys that; blindly preserving them can leave stale access when the configured platform ServiceAccount changes.

**Decision.** Extra subjects are preserved while the expected ServiceAccount is present; if it is absent, the subject list is reset to exactly the expected subject, with a warning listing what was removed.

**Consequence.** Day-to-day out-of-band grants survive, but a configuration change re-establishes a known-good binding. The reset is logged, not silent.

## 6. `pods/exec` stays in the platform role by default, behind a Helm gate

**Context.** Exec access lets platform users debug inside containers, but it also bypasses image-level controls and is unaudited unless the cluster configures audit policy. Some customers want it gone.

**Decision.** The grant is kept by default (preserving platform behaviour) and gated by a Helm value; disabling it removes only exec, leaving log/status access.

**Consequence.** Security-sensitive customers get a one-line hardening knob without forking the chart; the default remains permissive, which reviewers should note.
