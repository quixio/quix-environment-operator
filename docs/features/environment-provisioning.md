# Environment provisioning

<!-- solution-docs:begin feature-environment-provisioning -->
## What it does

Creating an `Environment` object with an ID yields a dedicated, operator-managed namespace. The namespace name is the ID plus a configurable suffix (default `-qdep`), so environments are recognisable and cannot collide with ordinary namespaces. The environment can also carry labels and annotations to propagate onto the namespace — useful for cost attribution, policy selectors, or tooling that discovers environments by label.

## Why it exists

The platform needs isolated spaces created on demand, with names and metadata it can rely on, on clusters where it has no permission to create namespaces itself. Making the request declarative (a CR, not an API call to the operator) means provisioning is retryable, auditable, and observable with plain `kubectl`.

## Constraints and guarantees

- **Strict naming.** Environment IDs must fit the operator’s accepted namespace-safe shape, and customers can add an extra installation-level restriction when they need tighter naming policy.
- **Immutability.** Neither the `Environment` name nor its ID can change after creation — renaming would orphan the provisioned namespace.
- **Guarded metadata.** Propagated labels/annotations must be `quix.io/`-prefixed, and the operator's own identity keys (`managed-by`, `environment-id`, `environment-name`, …) are protected from override — both at admission and during reconciliation.
- **Ownership before action.** The operator stamps identity labels on namespaces it creates and verifies them before any adoption or deletion. A name collision with an unmanaged or foreign namespace fails the environment; it never hijacks the namespace.
