# Scoped platform RBAC

<!-- solution-docs:begin feature-scoped-platform-rbac -->
## What it does

For every environment, the operator creates a RoleBinding in the managed namespace that binds the platform ServiceAccount to a pre-approved platform ClusterRole. The platform thereby gains its workload permissions one namespace at a time — never cluster-wide — and losing an environment automatically revokes the access that came with it.

## Why it exists

This is the core security promise to customers: the platform's reach is the sum of the namespaces it asked for, and the *content* of its permissions is fixed in a ClusterRole the customer reviewed at install time. The operator only decides **where** those permissions apply, never **what** they are.

## Constraints and guarantees

- **Escalation validation.** Before binding, the configured ClusterRole is checked for rules that would let the platform escalate its own privileges (RBAC write access and similar). Validation runs fail-fast at operator startup and again on every reconcile, so an out-of-band edit to the ClusterRole is caught.
- **Least-privilege operator.** The operator itself holds only the `bind` verb on the platform ClusterRole — not the workload permissions it grants (see [decisions.md](../decisions.md)).
- **Subject reconciliation contract.** Manually added extra subjects on a managed RoleBinding are tolerated *as long as the expected platform ServiceAccount is present*. If the expected subject is missing (e.g. the configured ServiceAccount changed), the subject list is reset to exactly the expected subject and a warning lists what was removed.
- **`pods/exec` is gated.** The platform ClusterRole's exec grant is controlled by a Helm value (`env.allowPodsExec`, default on); turning it off removes only exec while keeping log/status access for debugging.
- **The cluster-scoped exception.** `Environment` objects are cluster-scoped, so a namespaced RoleBinding cannot cover them; the platform gets a dedicated, narrow ClusterRoleBinding for `environments` only. This is the sole cluster-wide platform grant and is a documented, deliberate exception.

Code entry points: `internal/resources/rolebinding/`, `internal/security/`
